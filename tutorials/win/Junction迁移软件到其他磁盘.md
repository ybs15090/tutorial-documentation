# 用 Junction 把 C 盘软件/数据迁移到其他磁盘

## 是什么

**目录联接（Junction）** 是 Windows 文件系统内核级别的路径重定向。把数据搬到 D/E 盘后，在原 C 盘位置建一个 Junction，任何程序访问原路径都会被透明地转到新位置——程序完全感知不到，行为与迁移前完全一致。

与快捷方式（.lnk）的本质区别：

| | 快捷方式 | Junction |
|---|---|---|
| 程序能否透明访问 | 否，程序需自行解析 | 是，内核层透明穿透 |
| 占 C 盘空间 | 否（只是指针文件） | 否（数据物理在目标盘） |
| 需要管理员权限 | 否 | 否 |
| 适合迁移软件数据 | 不适合 | 适合 |

---

## 适用场景

- 软件强制安装在 C 盘，不给选安装位置
- 已经装在 C 盘，不想重装
- `AppData` 下各软件的缓存/用户数据目录
- Steam / Epic 游戏文件夹
- npm 全局包、pip 缓存、Docker 镜像等开发工具数据

---

## 操作步骤

### 第一步：完全关闭软件

确保目标软件没有在运行（任务管理器确认进程已退出），否则文件被锁，复制会失败或不完整。

### 第二步：搬移数据到目标盘

用 `robocopy` 搬（支持长路径、自动处理权限）：

```powershell
robocopy "C:\原始路径" "D:\目标路径" /E /MOVE /R:0 /W:0
```

参数说明：
- `/E` — 包含子目录（含空目录）
- `/MOVE` — 移动（复制完删源文件）
- `/R:0 /W:0` — 失败不重试，直接跳过

> **注意**：robocopy 退出码 `1` = 成功（有文件被复制），不是错误。只有 `>= 8` 才是真正出错。

### 第三步：删除原文件夹（现在应为空）

```cmd
rmdir "C:\原始路径"
```

如果 robocopy 搬完后原目录已变空目录，直接删即可。若不为空说明有文件被跳过，检查后再删。

### 第四步：在原位置建 Junction

```cmd
cmd /c mklink /J "C:\原始路径" "D:\目标路径"
```

> 在 PowerShell 里运行需要加 `cmd /c`，因为 `mklink` 是 cmd 内建命令。
> 在 cmd 窗口里可以直接写 `mklink /J ...`。

### 第五步：启动软件验证

正常启动软件，检查数据、设置、登录状态是否完整。一切正常则迁移完成。

---

## 验证 Junction 是否生效

```cmd
cmd /c dir /a:l "C:\原始路径的父目录"
```

看到类似这一行就说明 Junction 生效：

```
<JUNCTION>     软件文件夹名 [D:\目标路径]
```

也可以用 PowerShell：

```powershell
Get-Item "C:\原始路径" | Format-List LinkType, Target
```

输出 `LinkType: Junction`，`Target: D:\目标路径` 即正确。

---

## 注意事项

### 删除 Junction 时务必用 rmdir

```cmd
rmdir "C:\Junction路径"
```

**不要**用资源管理器右键删除，也不要用 `rd /s`——这两种方式会把 D 盘的真实数据一起删掉。

`rmdir`（不加 `/s`）只删 Junction 本身，D 盘数据完全不受影响。

### 自动更新可能覆盖 Junction

部分软件的自动更新程序会重建安装目录，可能覆盖 Junction。更新后如果软件出问题，检查原路径是否还是 Junction（用 `dir /a:l` 验证），若变成了普通文件夹，重新搬迁一次即可。

### 注册表路径问题

Junction 解决的是文件系统路径。极少数软件会在注册表里硬编码路径并直接操作（而非通过文件系统 API），这类软件迁移后可能报错。遇到问题先还原，再考虑其他方案。

### 不要动系统文件夹

`C:\Windows`、`C:\Windows\System32`、`C:\Program Files\Common Files` 等系统路径不要用此方法迁移，可能导致系统不稳定。

---

## 还原方法

1. 完全关闭软件
2. 删除 Junction：`rmdir "C:\Junction路径"`
3. 把数据搬回来：`robocopy "D:\目标路径" "C:\原始路径" /E /MOVE /R:0 /W:0`

还原后和迁移前完全一致。

---

## 常见错误

| 错误信息 | 原因 | 解决 |
|---|---|---|
| `文件已存在` | 原文件夹没删干净 | 确保原路径已删除再建 Junction |
| `拒绝访问` | 有进程占用文件 | 彻底关闭软件再操作 |
| robocopy 退出码 8+ | 有文件复制失败 | 查看 robocopy 输出，处理被锁文件 |
| Junction 变普通文件夹 | 软件更新重建了目录 | 重新搬迁 |

## 符号链接（Symbolic Link）

符号链接是比 Junction 更通用的路径重定向机制，Vista 引入，概念与 Linux 的 `ln -s` 相同。

### 与 Junction 的对比

| | Junction | 符号链接 |
|---|---|---|
| **目标类型** | 只能指向目录 | 目录或文件均可 |
| **跨主机/网络路径** | 不支持 | 支持（可指向 UNC 路径） |
| **路径类型** | 只能绝对路径 | 绝对或相对路径均可 |
| **需要管理员权限** | 否 | 是（默认）† |
| **Git 处理** | 当普通目录，透明 | 当特殊文件对象追踪 |
| **Linux/Mac 兼容** | 否，Windows 专有 | 是，三平台通用概念 |
| **老程序兼容性** | 最好 | 极少数老程序行为异常 |

† Windows 10 开启**开发者模式**（设置 → 隐私和安全性 → 开发者选项）后普通用户也可创建符号链接。

### 创建命令

```cmd
REM 目录符号链接
cmd /c mklink /D "C:\链接路径" "D:\目标目录"

REM 文件符号链接
cmd /c mklink "C:\链接文件.txt" "D:\目标文件.txt"
```

### 删除符号链接

与 Junction 相同规则——只删链接本身，不动目标数据：

```cmd
REM 目录符号链接
rmdir "C:\链接路径"

REM 文件符号链接
del "C:\链接文件.txt"
```

### 何时选符号链接而非 Junction

- 需要链接**单个文件**（Junction 做不到）
- 需要**相对路径**（便于整体迁移后链接仍有效）
- 需要指向**网络路径**（UNC）
- 在 **WSL / Linux 环境**下操作需要识别
- 需要被 **Git 追踪**链接本身（而非其内容）

**迁移本机软件数据到其他磁盘**这个场景，优先选 Junction——不需要管理员权限，兼容性更好，对 Git 完全透明。
